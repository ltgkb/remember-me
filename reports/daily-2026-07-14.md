## 遇到的问题与解决方案

### 问题 1: extract/search 端点初始返回 400

| 项目 | 内容 |
|------|------|
| **现象** | `POST /extract` 和 `POST /search` 连续 3 次返回 400 Bad Request |
| **根因** | 请求体 JSON 格式或必填字段与 `server.py` 解析逻辑不匹配（如 `text` 字段缺失、或 `Content-Type` 未设置） |
| **调试过程** | 02:03:39 → 02:05:20 共 3 次 400；02:06:11 修复后 200 |
| **修复方案** | 在 `test_endpoints.py` 中显式设置 `Content-Type: application/json`，确保请求体包含必填字段（`text` 用于 extract，`keyword` 用于 search） |
| **验证结果** | 最终 extract 200（返回 2 条结果）、search 200（返回 0 条匹配但字段结构正确） |

### 问题 2: ProjectManager.list 排序毫秒级冲突

| 项目 | 内容 |
|------|------|
| **现象** | `list 应按更新时间倒序返回项目` 测试因时间戳相同导致排序不稳定 |
| **根因** | `create()` 使用 `new Date().toISOString()`，两个项目在同一毫秒内创建时 `updatedAt` 相同 |
| **修复方案** | 测试用例改为 `async`，在两次 `create` 之间插入 `await new Promise(r => setTimeout(r, 2))`，确保时间戳严格递增 |
| **验证结果** | 测试稳定通过，编译产物已更新（`out/test/suite/project.test.js` 时间戳 02:04） |

### 问题 3: Windows 目录删除 EPERM

| 项目 | 内容 |
|------|------|
| **现象** | `ProjectManager.delete()` 测试日志中出现 `EPERM: operation not permitted, unlink`（针对 `.backups` 和 `conversations` 子目录） |
| **根因** | `JsonStorage.delete()` 使用 `fs.unlinkSync`，无法删除目录，Windows 下返回 `EPERM` |
| **影响** | 测试本身通过（`afterEach` 清理临时目录），但生产环境可能残留空子目录 |
| **解决方案** | 已记录于 `e2e-issues-2026-07-13.md`，建议后续改用 `fs.rmSync(path, { recursive: true })` 或增加目录类型判断 |
