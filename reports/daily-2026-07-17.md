### 2. A1 扩展端轮询逻辑修复（extension.ts）✅

`packages/vscode-extension/src/extension.ts` 语义模型预热轮询做了 4 处健壮性修复：

- **内存泄漏修复**：`semanticPollInterval` 提升为模块级变量，`deactivate()` 中显式 `clearInterval`；
- **竞态修复**：超时检查移入 `healthCheck` 异步回调内，消除 `pollCount` 与 interval 状态不同步；

## 遇到的问题

### 问题 #1（P0）：Python 3.14 环境下 ChromaDB 初始化失败，且未优雅降级

**现象**（08:06 实测，`.venv` Python 3.14.3 + chromadb 1.5.9）：

1. 后台预加载线程失败：`[preload] 后台向量索引预加载失败 (AttributeError): 'RustBindingsAPI' object has no attribute 'bindings'` —— chromadb 1.5.9 的 Rust 绑定在 Python 3.14 上不兼容；
2. 预加载失败后，`SharedSystemClient` 注册表残留不一致状态，请求线程再次 `PersistentClient` 时抛 `KeyError: 'C:\Users\linbi\.remember-me\vector_db'`；
3. **该 KeyError 直接穿透 HTTP 处理器**：`vector_index._ensure_initialized()` 只把 `ImportError` 包装为 `SemanticSearchError`，运行时异常原样上抛；`get_vector_index()` 也只捕获 `SemanticSearchError` —— 导致连接被重置（`RemoteDisconnected`），而非设计中的 503 优雅降级。

**影响**：当前默认 `.venv`（Python 3.14）下语义搜索 / 混合搜索端点全部不可用；首次查询延迟 <200ms 的验收目标无法测量。

**环境疑点**：`__pycache__` 显示凌晨会话使用 **Python 3.12** 运行（`server.cpython-312.pyc`，02:05），而 `scripts/` 与默认 `.venv` 为 3.14 —— 凌晨会话可能未在 3.14 下复验，两套解释器并存本身即是风险。

### 建议解决方案

- **短期（代码兜底，~10 行）**：`_ensure_initialized()` 将 `chromadb.PersistentClient` 调用包入 `try/except Exception → SemanticSearchError`；`get_vector_index()` 同步兜底。保证任何环境下语义栈故障都走 503 降级而非处理器崩溃；
- **环境固定（二选一）**：a) `pyproject.toml` 声明 `requires-python = ">=3.11,<3.14"`，统一使用与凌晨会话一致的 Python 3.12；b) 升级 chromadb 至兼容 3.14 的版本（需查其 release notes 确认）；
- **CI 覆盖**：B1 执行时将 Python 矩阵纳入 3.12 + 3.14，让此类问题在 CI 暴露而非本地。
