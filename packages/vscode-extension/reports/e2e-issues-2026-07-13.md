## 任务 C1：修复 ProjectManager.list 排序测试失败

### 问题分析
测试用例 `list 应按更新时间倒序返回项目` 在连续调用 `manager.create('ProjectA')` 和 `manager.create('ProjectB')` 时，由于两次创建可能在同一毫秒内完成，导致 `updatedAt` 和 `createdAt` 时间戳完全相同。此时三级排序逻辑（updatedAt 倒序 → createdAt 倒序 → name 正序）会按 `name` 正序排列，'ProjectA' < 'ProjectB'，因此 `list[0]` 为 'ProjectA'，与测试断言 `list[0].name === 'ProjectB'` 矛盾。

### 修复方案
**修改文件**：`src/test/suite/project.test.ts`（第 75-82 行）

将测试用例改为 `async`，在两个 `create` 调用之间插入 `await new Promise(r => setTimeout(r, 2))`，确保 `ProjectB` 的 `updatedAt` 严格晚于 `ProjectA`：

```typescript
it('list 应按更新时间倒序返回项目', async () => {
  manager.create('ProjectA', 'A', 'B');
  await new Promise(r => setTimeout(r, 2));
  manager.create('ProjectB', 'C', 'D');
  const list = manager.list();
  assert.strictEqual(list.length, 2);
  assert.strictEqual(list[0].name, 'ProjectB');
});
```

### 修复结果
✅ 测试稳定通过，不再受毫秒级时间精度影响。

## 其他发现（非阻塞，记录备查）

### 1. ProjectManager.delete 在 Windows 上产生 EPERM 日志
在运行 `delete 应删除项目` 测试时，日志中出现两条 `EPERM` 错误：

```
[ERROR] 删除文件失败: ...\.backups    {errno:-4048, code:EPERM, syscall:unlink}
[ERROR] 删除文件失败: ...\conversations {errno:-4048, code:EPERM, syscall:unlink}
```

**根因**：`ProjectManager.delete()` 先调用 `storage.listDir('projects', safeName)` 列出项目目录下的所有条目，然后逐个调用 `storage.delete()`。但 `listDir` 会返回子目录名（如 `conversations`、`.backups`），而 `JsonStorage.delete()` 底层使用 `fs.unlinkSync`，无法删除目录，导致在 Windows 上返回 `EPERM`。

**影响**：测试本身通过，因为 `afterEach` 会清理整个临时目录；但生产环境下可能导致项目目录残留空子目录。

**建议**：在 `ProjectManager.delete()` 或 `JsonStorage.delete()` 中增加对目录的递归删除处理，或改用 `fs.rmSync(path, { recursive: true })` 删除整个项目目录。
