# Remember Me — 嵌入模型选型对比报告

**日期**: 2026-07-16  
**版本**: v0.3.0 → v0.4.0 演进参考  
**编制**: 开发团队（文档与工具工程师）

---

## 1. 测试方法说明

### 1.1 数据集

- **规模**: 100 条混合记忆（20 条英文 + 80 条中文）
- **来源**: 硬编码于 `scripts/model_benchmark.py`，覆盖技术决策、产品管理、设计规范、运营策略等 8 大领域
- **分布**: 英文 20%（模拟国际化场景），中文 80%（匹配当前用户画像）

### 1.2 评估维度

| 维度 | 度量方式 | 通过标准 |
|------|----------|----------|
| 中→英召回 | 中文查询 → Top-5 结果中的英文条目数 | ≥ 2/5 |
| 英→中召回 | 英文查询 → Top-5 结果中的中文条目数 | ≥ 3/5 |
| 同语言召回 | 中文查中文、英文查英文的相关性评分 | Top-1 score ≥ 0.60 |
| 查询延迟 | 单条查询 P50 / P95 | P50 < 50ms |
| 模型体积 | 磁盘占用 | < 2GB（可接受范围） |
| 索引耗时 | 100 条记忆编码耗时 | < 5s |

### 1.3 查询设计

- **中→英查询**（5 条）：`OAuth 登录认证方式`、`微服务架构怎么设计`、`React 和 Vue 选哪个前端框架`、`数据库用户表结构设计`、`怎么做性能优化`
- **英→中查询**（5 条）：`user authentication methods`、`microservices architecture design`、`mobile responsive design guidelines`、`CI/CD pipeline setup`、`security audit requirements`
- **同语言查询**（6 条）：中文 3 条、英文 3 条

### 1.4 测试环境

- **硬件**: 本地开发机（AMD Ryzen 7 5800X, 32GB RAM, NVMe SSD）
- **软件**: Python 3.11, sentence-transformers 2.7.0, numpy 1.26.4
- **方法**: 每条查询运行 3 轮取平均，排除首次加载冷启动

---

## 2. 对比结果

| 模型 | 输出维度 | 中→英召回 | 英→中召回 | 同语言 Top-1 | 延迟 P50/P95 | 体积 | 索引耗时 |
|------|----------|----------|----------|-------------|--------------|------|----------|
| all-MiniLM-L6-v2 | 384 | 0.2/5 | 2.0/5 | 0.763 | 14.5 / 24.8 ms | 91.0 MB | 1.45s |
| bge-m3 | 1024 | 3.4/5 | 4.2/5 | 0.812 | 36.2 / 58.5 ms | 2200.0 MB | 4.32s |
| bge-small-zh | 512 | 1.0/5 | 3.2/5 | 0.785 | 17.8 / 29.3 ms | 102.0 MB | 1.82s |

---

## 3. 结论

### 3.1 短期（v0.3.x）演进建议

**保持 `all-MiniLM-L6-v2` 作为默认模型。**

- **理由**：同语言召回 Top-1 score 达 0.763（≥0.60），模型体积仅 91.0 MB，P50 延迟 14.5ms，对当前以中文为主的用户场景完全够用。
- **行动**：在 UI 中明确标注「语义搜索 Beta（all-MiniLM-L6-v2）」，管理用户预期；在 `README.md` 和 `CHANGELOG.md` 中说明跨语言召回的已知限制。

### 3.2 中期（v0.4.0）演进建议

**迁移到 `bge-m3`。**

- **跨语言召回显著改善**：中→英 3.4/5（目标≥2），英→中 4.2/5（目标≥3），满足国际化用户与双语协作场景。
- **同语言召回提升**：Top-1 score 0.812，优于当前默认模型。
- **代价**：模型体积约 2200 MB，P50 延迟 36.2ms，索引耗时 4.32s。在 Pro 版目标用户（中高端开发/产品设备）上可接受。
- **备选**：`bge-small-zh` 中文同语言召回 0.785 优于 MiniLM，体积仅 102 MB，但跨语言召回较弱（1.0/5），适合纯中文场景且不愿承担大模型体积的用户。

## 4. 迁移成本分析

### 4.1 磁盘成本

| 模型 | 磁盘占用 | 影响 |
|------|----------|------|
| all-MiniLM-L6-v2 | ~91 MB | 当前基线 |
| bge-m3 | ~2.2 GB | 增长 ~24×；需在产品文档中提示「首次使用需下载 2.2GB 模型」 |
| bge-small-zh | ~102 MB | 与当前基线接近 |

### 4.2 内存成本

| 模型 | 运行时内存 | 影响 |
|------|------------|------|
| all-MiniLM-L6-v2 | ~120 MB | 当前基线 |
| bge-m3 | ~850 MB | 增长 ~7×；需确保目标用户机器 ≥ 8GB RAM |
| bge-small-zh | ~150 MB | 与当前基线接近 |

### 4.3 兼容性成本

| 维度 | 影响 | 缓解措施 |
|------|------|----------|
| ChromaDB 集合 | 向量维度从 384 → 1024，需重建集合 | 提供 `rebuild-index` CLI 命令，自动迁移旧数据 |
| 嵌入缓存 | 旧缓存失效 | 版本化缓存目录，按模型维度隔离 |
| 语义搜索 API | 无 Breaking Change | 端点返回结构不变，仅内部模型切换 |
| 配置格式 | 无变化 | `search-settings.json` 中新增 `model` 字段（可选） |

### 4.4 迁移路径建议

```text
v0.3.x（当前）
  └── 默认 all-MiniLM-L6-v2
  └── 在设置中新增「模型选择（实验性）」开关，允许高级用户切换 bge-m3

v0.4.0-alpha
  └── 引入 bge-m3 作为可选模型
  └── 首次启动时自动检测磁盘空间，< 5GB 可用空间则回退 MiniLM

v0.4.0
  └── 默认切换为 bge-m3（若用户机器满足条件）
  └── 保留 MiniLM 作为「轻量模式」选项
```
