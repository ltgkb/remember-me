# 语义搜索快速原型验证报告

**任务编号**: D1
**验证日期**: 2026-07-14
**验证人**: AI 适配与引擎验证子代理
**原型脚本**: `packages/memory-engine/scripts/semantic_search_prototype.py`

---

## 1. 原型概述

本原型用于验证 **ChromaDB + sentence-transformers** 在 Remember Me 项目中实现语义搜索的技术可行性，重点评估：

1. 嵌入模型加载与推理性能
2. 向量索引构建效率
3. 查询延迟（Top-5 检索）
4. 跨语言语义匹配效果（中 → 英、英 → 中、混合）
5. 磁盘占用

---

## 2. 技术栈与配置

| 组件 | 版本/型号 | 说明 |
|------|----------|------|
| Python | 3.14.3 | 64-bit Windows |
| 嵌入模型 | `all-MiniLM-L6-v2` | 384 维，多语言支持，Hugging Face 官方模型 |
| 向量数据库 | ChromaDB (PersistentClient) | 使用 cosine 空间度量 |
| 备选方案 | NumPy cosine similarity | 当 ChromaDB/sentence-transformers 不可用时自动降级 |
| 数据规模 | 100 条模拟记忆片段 | 中英文混合，覆盖 5 个业务类别 |

## 4. 性能基准测试结果

### 4.1 模型加载

| 指标 | 数值 |
|------|------|
| 模型加载耗时 | **7,173.68 ms** (~7.2 s) |
| 模型来源 | Hugging Face Hub (首次下载后本地缓存) |
| 模型体积 | ~80 MB (all-MiniLM-L6-v2) |

> **说明**: 模型加载为一次性开销，服务启动后常驻内存。建议在生产环境中预加载模型，避免用户首次查询时的冷启动延迟。

### 4.2 索引构建

| 指标 | 数值 |
|------|------|
| 100 条记忆编码耗时 | **853.46 ms** |
| 单条平均编码耗时 | ~8.5 ms |
| ChromaDB 索引构建耗时 | **87.59 ms** |
| 磁盘占用 (持久化) | **528.0 KB** (100 条) |

> **估算**: 若按 10,000 条记忆估算，编码约 85 s，磁盘占用约 50–60 MB（含 HNSW 索引开销）。

### 4.3 查询延迟 (Top-5)

| 查询语言 | 查询内容 | 延迟 |
|---------|---------|------|
| 中文 | "用户登录相关的讨论" | **11.83 ms** |
| 英文 | "authentication and OAuth decisions" | **1.58 ms** |
| 混合 | "Python 项目的认证方案" | **1.61 ms** |

> **观察**: 中文查询延迟略高，可能与 sentence-transformers 对中文 tokenization 的处理有关，但整体均远低于 100 ms 的目标阈值。

## 5. 语义召回质量分析

### 5.1 中文查询："用户登录相关的讨论"

| 排名 | 分数 | 语言 | 召回内容摘要 |
|------|------|------|-------------|
| #1 | 0.6819 | zh | 与设计师沟通搜索结果页需展示相似度分数和匹配关键词高亮 |
| #2 | 0.6737 | zh | 产品评审确认语义搜索默认排序按相似度降序，支持按时间二次排序 |
| #3 | 0.5921 | zh | 团队讨论搜索结果的「为什么匹配」解释功能，先用关键词重叠高亮做 MVP |
| #4 | 0.5776 | zh | 竞品缺陷：大多数笔记工具不支持向量搜索，只依赖标签和文件夹分类 |
| #5 | 0.5767 | zh | 竞品观察：Logseq 的查询语法强大但太复杂，普通用户难以掌握 |

**分析**: Top-5 全部召回中文记忆，但内容偏向"搜索功能设计"而非"用户登录"。all-MiniLM-L6-v2 对中文语义的理解有限，"登录"与"搜索"在向量空间中距离较近，说明通用模型对垂直领域（开发者工具）的语义区分度不足。

### 5.2 英文查询："authentication and OAuth decisions"

| 排名 | 分数 | 语言 | 召回内容摘要 |
|------|------|------|-------------|
| #1 | 0.4355 | zh | PRD v2.3：用户登录模块需支持 OAuth2 + 短信验证码双因子认证 |
| #2 | 0.3298 | en | Security review: API tokens must be rotated every 90 days |
| #3 | 0.2775 | en | PRD requirement: login flow must support SSO integration with Azure AD |
| #4 | 0.2090 | en | Architecture decision: use FastAPI for memory-engine HTTP service |

**分析**: #1 正确召回中文记忆（"OAuth2 + 短信验证码"），实现了**跨语言语义匹配**。但总体分数偏低（最高 0.4355），说明英文查询对中文内容的召回能力存在但不够强。英文记忆 (#2, #3) 也合理，但分数分散。

## 6. 跨语言语义匹配评估

| 查询 | 期望召回语言 | 实际召回数量 (Top-5) | 评估 |
|------|------------|---------------------|------|
| 中文查询 "用户登录相关的讨论" | 英文 | **0 / 5** | 跨语言召回失败 |
| 英文查询 "authentication and OAuth decisions" | 中文 | **2 / 5** | 部分跨语言召回成功 |
| 混合查询 "Python 项目的认证方案" | 英文 | **0 / 5** | 跨语言召回失败 |

**结论**: all-MiniLM-L6-v2 作为通用多语言模型，在**英→中**方向有一定跨语言匹配能力（2/5），但在**中→英**方向表现较弱。这与模型训练语料中中英文对齐质量有关。若需更强的跨语言检索，建议：

1. **微调方案**: 在领域语料（开发者文档、技术讨论）上微调嵌入模型
2. **替代模型**: 评估 `bge-m3` 或 `bge-small-zh` 对中英文混合场景的适配性
3. **翻译增强**: 查询时自动将中文翻译为英文后并行检索，合并结果

## 7. 磁盘占用评估

| 指标 | 数值 |
|------|------|
| 100 条记忆持久化大小 | **528.0 KB** |
| 单条平均占用 | ~5.3 KB |
| 10,000 条预估 | ~50–60 MB |
| 100,000 条预估 | ~500–600 MB |

> ChromaDB 使用 SQLite 作为底层存储，包含向量数据、HNSW 索引和元数据。随着数据量增长，可考虑：
> - 定期清理过期记忆的向量索引
> - 对旧数据降采样或归档

## 8. 风险与建议

### 8.1 技术风险

| 风险 | 严重度 | 说明 | 缓解措施 |
|------|--------|------|---------|
| 模型冷启动延迟 (~7s) | Medium | 首次加载模型影响用户体验 | 服务启动时预加载；提供加载进度提示 |
| 跨语言召回不一致 | Medium | 中→英召回弱于英→中 | 评估 bge-m3；考虑双语查询扩展 |
| 通用模型领域适配不足 | Medium | 对技术术语语义区分度有限 | 收集领域语料微调；或使用代码专用嵌入模型 |
| ChromaDB Windows 兼容性 | Low | 原型验证通过，但需长期观察 | 持续监控；保留 NumPy 降级方案 |

### 8.2 后续行动建议

1. **模型选型对比**: 在同等数据集上测试 `bge-small-zh`、`bge-m3`、`text-embedding-3-small` 的召回质量与延迟
2. **微调实验**: 使用 500+ 条真实记忆数据对 all-MiniLM-L6-v2 进行领域微调
3. **混合检索**: 结合关键词搜索（当前已实现）与语义搜索，实现 BM25 + 向量的混合排序
4. **ONNX 推理优化**: 原型日志显示 ONNX Runtime 可带来 2x 加速，评估生产环境集成成本

| 检查项 | 结果 |
|--------|------|
